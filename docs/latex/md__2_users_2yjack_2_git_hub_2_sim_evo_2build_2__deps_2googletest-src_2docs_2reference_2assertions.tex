 This page lists the assertion macros provided by Google\+Test for verifying code behavior. To use them, include the header {\ttfamily \doxylink{gtest_8h_source}{gtest/gtest.\+h}}.

The majority of the macros listed below come as a pair with an {\ttfamily EXPECT\+\_\+} variant and an {\ttfamily ASSERT\+\_\+} variant. Upon failure, {\ttfamily EXPECT\+\_\+} macros generate nonfatal failures and allow the current function to continue running, while {\ttfamily ASSERT\+\_\+} macros generate fatal failures and abort the current function.

All assertion macros support streaming a custom failure message into them with the {\ttfamily \texorpdfstring{$<$}{<}\texorpdfstring{$<$}{<}} operator, for example\+:


\begin{DoxyCode}{0}
\DoxyCodeLine{EXPECT\_TRUE(my\_condition)\ <<\ \textcolor{stringliteral}{"{}My\ condition\ is\ not\ true"{}};}

\end{DoxyCode}


Anything that can be streamed to an {\ttfamily ostream} can be streamed to an assertion macro—in particular, C strings and string objects. If a wide string ({\ttfamily wchar\+\_\+t\texorpdfstring{$\ast$}{*}}, {\ttfamily TCHAR\texorpdfstring{$\ast$}{*}} in {\ttfamily UNICODE} mode on Windows, or {\ttfamily std\+::wstring}) is streamed to an assertion, it will be translated to UTF-\/8 when printed.\hypertarget{md__2_users_2yjack_2_git_hub_2_sim_evo_2build_2__deps_2googletest-src_2docs_2reference_2assertions_success-failure}{}\doxysection{\texorpdfstring{Explicit Success and Failure}{Explicit Success and Failure}}\label{md__2_users_2yjack_2_git_hub_2_sim_evo_2build_2__deps_2googletest-src_2docs_2reference_2assertions_success-failure}
The assertions in this section generate a success or failure directly instead of testing a value or expression. These are useful when control flow, rather than a Boolean expression, determines the test\textquotesingle{}s success or failure, as shown by the following example\+:


\begin{DoxyCode}{0}
\DoxyCodeLine{\ ++}
\DoxyCodeLine{\textcolor{keywordflow}{switch}(expression)\ \{}
\DoxyCodeLine{\ \ \textcolor{keywordflow}{case}\ 1:}
\DoxyCodeLine{\ \ \ \ ...\ some\ checks\ ...}
\DoxyCodeLine{\ \ \textcolor{keywordflow}{case}\ 2:}
\DoxyCodeLine{\ \ \ \ ...\ some\ other\ checks\ ...}
\DoxyCodeLine{\ \ \textcolor{keywordflow}{default}:}
\DoxyCodeLine{\ \ \ \ FAIL()\ <<\ \textcolor{stringliteral}{"{}We\ shouldn't\ get\ here."{}};}
\DoxyCodeLine{\}}

\end{DoxyCode}

{\ttfamily SUCCEED()}

Generates a success. This {\itshape does not} make the overall test succeed. A test is considered successful only if none of its assertions fail during its execution.

The {\ttfamily SUCCEED} assertion is purely documentary and currently doesn\textquotesingle{}t generate any user-\/visible output. However, we may add {\ttfamily SUCCEED} messages to Google\+Test output in the future.\hypertarget{md__2_users_2yjack_2_git_hub_2_sim_evo_2build_2__deps_2googletest-src_2docs_2reference_2assertions_FAIL}{}\doxysubsection{\texorpdfstring{FAIL}{FAIL}}\label{md__2_users_2yjack_2_git_hub_2_sim_evo_2build_2__deps_2googletest-src_2docs_2reference_2assertions_FAIL}
{\ttfamily FAIL()}

Generates a fatal failure, which returns from the current function.

Can only be used in functions that return {\ttfamily void}. Generates a nonfatal failure, which allows the current function to continue running.\hypertarget{md__2_users_2yjack_2_git_hub_2_sim_evo_2build_2__deps_2googletest-src_2docs_2reference_2assertions_ADD_FAILURE_AT}{}\doxysubsection{\texorpdfstring{ADD\+\_\+\+FAILURE\+\_\+\+AT}{ADD\+\_\+\+FAILURE\+\_\+\+AT}}\label{md__2_users_2yjack_2_git_hub_2_sim_evo_2build_2__deps_2googletest-src_2docs_2reference_2assertions_ADD_FAILURE_AT}
{\ttfamily ADD\+\_\+\+FAILURE\+\_\+\+AT(}\texorpdfstring{$\ast$}{*}{\ttfamily file\+\_\+path}\texorpdfstring{$\ast$}{*}{\ttfamily ,}\texorpdfstring{$\ast$}{*}{\ttfamily line\+\_\+number}\texorpdfstring{$\ast$}{*}{\ttfamily )}

Generates a nonfatal failure at the file and line number specified.\hypertarget{md__2_users_2yjack_2_git_hub_2_sim_evo_2build_2__deps_2googletest-src_2docs_2reference_2assertions_generalized}{}\doxysection{\texorpdfstring{Generalized Assertion}{Generalized Assertion}}\label{md__2_users_2yjack_2_git_hub_2_sim_evo_2build_2__deps_2googletest-src_2docs_2reference_2assertions_generalized}
The following assertion allows \doxysectlink{md__2_users_2yjack_2_git_hub_2_sim_evo_2build_2__deps_2googletest-src_2docs_2reference_2matchers}{matchers}{0} to be used to verify values.\hypertarget{md__2_users_2yjack_2_git_hub_2_sim_evo_2build_2__deps_2googletest-src_2docs_2reference_2assertions_EXPECT_THAT}{}\doxysubsection{\texorpdfstring{EXPECT\+\_\+\+THAT}{EXPECT\+\_\+\+THAT}}\label{md__2_users_2yjack_2_git_hub_2_sim_evo_2build_2__deps_2googletest-src_2docs_2reference_2assertions_EXPECT_THAT}
{\ttfamily EXPECT\+\_\+\+THAT(}\texorpdfstring{$\ast$}{*}{\ttfamily value}\texorpdfstring{$\ast$}{*}{\ttfamily ,}\texorpdfstring{$\ast$}{*}{\ttfamily matcher}\texorpdfstring{$\ast$}{*}{\ttfamily )} \textbackslash{} {\ttfamily ASSERT\+\_\+\+THAT(}\texorpdfstring{$\ast$}{*}{\ttfamily value}\texorpdfstring{$\ast$}{*}{\ttfamily ,}\texorpdfstring{$\ast$}{*}{\ttfamily matcher}\texorpdfstring{$\ast$}{*}{\ttfamily )}

Verifies that \texorpdfstring{$\ast$}{*}{\ttfamily value}\texorpdfstring{$\ast$}{*} matches the \doxysectlink{md__2_users_2yjack_2_git_hub_2_sim_evo_2build_2__deps_2googletest-src_2docs_2reference_2matchers}{matcher}{0} \texorpdfstring{$\ast$}{*}{\ttfamily matcher}\texorpdfstring{$\ast$}{*}.

For example, the following code verifies that the string {\ttfamily value1} starts with {\ttfamily "{}\+Hello"{}}, {\ttfamily value2} matches a regular expression, and {\ttfamily value3} is between 5 and 10\+:


\begin{DoxyCode}{0}
\DoxyCodeLine{\textcolor{preprocessor}{\#include\ "{}gmock/gmock.h"{}}}
\DoxyCodeLine{}
\DoxyCodeLine{using\ ::testing::AllOf;}
\DoxyCodeLine{using\ ::testing::Gt;}
\DoxyCodeLine{using\ ::testing::Lt;}
\DoxyCodeLine{using\ ::testing::MatchesRegex;}
\DoxyCodeLine{using\ ::testing::StartsWith;}
\DoxyCodeLine{}
\DoxyCodeLine{...}
\DoxyCodeLine{EXPECT\_THAT(value1,\ StartsWith(\textcolor{stringliteral}{"{}Hello"{}}));}
\DoxyCodeLine{EXPECT\_THAT(value2,\ MatchesRegex(\textcolor{stringliteral}{"{}Line\ \(\backslash\)\(\backslash\)d+"{}}));}
\DoxyCodeLine{ASSERT\_THAT(value3,\ AllOf(Gt(5),\ Lt(10)));}

\end{DoxyCode}


Matchers enable assertions of this form to read like English and generate informative failure messages. For example, if the above assertion on {\ttfamily value1} fails, the resulting message will be similar to the following\+:


\begin{DoxyCode}{0}
\DoxyCodeLine{Value\ of:\ value1}
\DoxyCodeLine{\ \ Actual:\ "{}Hi,\ world!"{}}
\DoxyCodeLine{Expected:\ starts\ with\ "{}Hello"{}}

\end{DoxyCode}


Google\+Test provides a built-\/in library of matchers—see the \doxysectlink{md__2_users_2yjack_2_git_hub_2_sim_evo_2build_2__deps_2googletest-src_2docs_2reference_2matchers}{Matchers Reference}{0}. It is also possible to write your own matchers—see \href{../gmock_cook_book.md\#NewMatchers}{\texttt{ Writing New Matchers Quickly}}. The use of matchers makes {\ttfamily EXPECT\+\_\+\+THAT} a powerful, extensible assertion.

The following assertions compare two values. The value arguments must be comparable by the assertion\textquotesingle{}s comparison operator, otherwise a compiler error will result.

If an argument supports the {\ttfamily \texorpdfstring{$<$}{<}\texorpdfstring{$<$}{<}} operator, it will be called to print the argument when the assertion fails. Otherwise, Google\+Test will attempt to print them in the best way it can—see \href{../advanced.md\#teaching-googletest-how-to-print-your-values}{\texttt{ Teaching Google\+Test How to Print Your Values}}.

Arguments are always evaluated exactly once, so it\textquotesingle{}s OK for the arguments to have side effects. However, the argument evaluation order is undefined and programs should not depend on any particular argument evaluation order.

These assertions work with both narrow and wide string objects ({\ttfamily string} and {\ttfamily wstring}).

Verifies that the predicate \texorpdfstring{$\ast$}{*}{\ttfamily pred}\texorpdfstring{$\ast$}{*} returns {\ttfamily true} when passed the given values as arguments.

The parameter \texorpdfstring{$\ast$}{*}{\ttfamily pred}\texorpdfstring{$\ast$}{*} is a function or functor that accepts as many arguments as the corresponding macro accepts values. If \texorpdfstring{$\ast$}{*}{\ttfamily pred}\texorpdfstring{$\ast$}{*} returns {\ttfamily true} for the given arguments, the assertion succeeds, otherwise the assertion fails.

When the assertion fails, it prints the value of each argument. Arguments are always evaluated exactly once.

As an example, see the following code\+:


\begin{DoxyCode}{0}
\DoxyCodeLine{\textcolor{comment}{//\ Returns\ true\ if\ m\ and\ n\ have\ no\ common\ divisors\ except\ 1.}}

\DoxyCodeLine{const\ \textcolor{keywordtype}{int}\ a\ =\ 3;}
\DoxyCodeLine{\textcolor{keyword}{const}\ \textcolor{keywordtype}{int}\ b\ =\ 4;}
\DoxyCodeLine{\textcolor{keyword}{const}\ \textcolor{keywordtype}{int}\ c\ =\ 10;}
\DoxyCodeLine{...}
\DoxyCodeLine{EXPECT\_PRED2(MutuallyPrime,\ a,\ b);\ \ \textcolor{comment}{//\ Succeeds}}
\DoxyCodeLine{EXPECT\_PRED2(MutuallyPrime,\ b,\ c);\ \ \textcolor{comment}{//\ Fails}}

\end{DoxyCode}

\begin{DoxyCode}{0}
\DoxyCodeLine{MutuallyPrime(b,\ c)\ is\ false,\ where}
\DoxyCodeLine{b\ is\ 4}
\DoxyCodeLine{c\ is\ 10}

\end{DoxyCode}


Note that if the given predicate is an overloaded function or a function template, the assertion macro might not be able to determine which version to use, and it might be necessary to explicitly specify the type of the function. The following assertions verify that a piece of code causes the process to terminate. For context, see \href{../advanced.md\#death-tests}{\texttt{ Death Tests}}.

These assertions spawn a new process and execute the code under test in that process. How that happens depends on the platform and the variable {\ttfamily testing\+::\+GTEST\+\_\+\+FLAG(death\+\_\+test\+\_\+style)}, which is initialized from the command-\/line flag {\ttfamily -\/-\/gtest\+\_\+death\+\_\+test\+\_\+style}.


\begin{DoxyItemize}
\item On POSIX systems, {\ttfamily fork()} (or {\ttfamily clone()} on Linux) is used to spawn the child, after which\+:
\begin{DoxyItemize}
\item If the variable\textquotesingle{}s value is {\ttfamily "{}fast"{}}, the death test statement is immediately executed.
\item If the variable\textquotesingle{}s value is {\ttfamily "{}threadsafe"{}}, the child process re-\/executes the unit test binary just as it was originally invoked, but with some extra flags to cause just the single death test under consideration to be run.
\end{DoxyItemize}
\item On Windows, the child is spawned using the {\ttfamily Create\+Process()} API, and re-\/executes the binary to cause just the single death test under consideration to be run -\/ much like the {\ttfamily "{}threadsafe"{}} mode on POSIX.
\end{DoxyItemize}

Other values for the variable are illegal and will cause the death test to fail. Currently, the flag\textquotesingle{}s default value is \texorpdfstring{$\ast$}{*}\texorpdfstring{$\ast$}{*}{\ttfamily "{}fast"{}}\texorpdfstring{$\ast$}{*}\texorpdfstring{$\ast$}{*}.

If the death test statement runs to completion without dying, the child process will nonetheless terminate, and the assertion fails.
